IV.2  WHAT THE PIPELINE DOES NOT YET DO
-----------------------------------------
The current mine_chats.py is a first-pass keyword-matching pipeline.
It is good at breadth but weak at depth. Specifically:

  a) It does NOT extract the actual table structures the AI produced.
     We know you asked for 547 tables. We do not yet know what columns
     those tables actually had, what was in them, or which ones were
     most successful. The assistant responses are in the DB (role='assistant')
     but unanalysed.

  b) It does NOT identify which PDF documents you were working with.
     Tool turns contain embedded PDF text, but we have not extracted
     document titles, authors, or page ranges from them. We don't know
     which specific academic articles generated which conversations.

  c) It does NOT track refinement chains.
     We know 240 turns are refinements, but we haven't linked each
     refinement turn to the original request it modified, or tracked
     how many iterations a given table went through before you were satisfied.

  d) It does NOT extract or classify your prompt *templates*.
     We've inferred templates from analysis, but we haven't automatically
     clustered your prompts to discover templates you use that we haven't
     named yet.

  e) It does NOT use LLM-quality semantic understanding.
     All entity extraction uses keyword matching with word boundaries.
     This misses entities not in the dictionary, misclassifies ambiguous
     references, and cannot infer thematic context. An LLM-based pass
     would catch far more.

  f) It does NOT connect to your PDF collection.
     The ~2.8 million words of embedded tool-turn text is your PDF
     collection, partially. But we haven't mapped which PDF files
     correspond to which conversations, or built a unified bibliography.

────────────────────────────────────────────────────────────────────────────────
SCRIPT 1: extract_tables.py
Purpose:  Extract the actual table structures from assistant responses
          and save them to the DB and to tables.json for EsoPipe
────────────────────────────────────────────────────────────────────────────────

What it does:
  Scans assistant turns for Markdown table syntax (lines beginning with |).
  Parses the header row to extract column names. Parses data rows.
  Stores extracted tables in a new DB table: extracted_tables.
  Exports the best examples as JSON for EsoPipe.

Why it matters:
  We have 547 table requests but zero knowledge of what was actually
  produced. The column names in real AI-generated tables are the ground
  truth of what actually worked — not just what you asked for.

DB schema additions:
  CREATE TABLE extracted_tables (
    id              INTEGER PRIMARY KEY,
    conversation_id INTEGER REFERENCES conversations(id),
    turn_id         INTEGER REFERENCES turns(id),
    preceding_request TEXT,  -- the user turn that requested this table
    column_names    TEXT,    -- JSON array of column headers
    row_count       INTEGER,
    raw_markdown    TEXT
  );

Parsing approach:
  Look for consecutive lines starting with | in assistant turns.
  A table is: header_row | separator_row | data_rows+
  Use regex: r'^\|(.+)\|' to match table lines.
  The header row gives column names. Separator row (---|---) confirms
  it's a table, not just a quoted block.

Output:
  - extracted_tables table in DB with all parsed tables
  - column_name_frequency.csv: ranked list of all column names ever used
  - tables_real.json: the 20 richest tables, ready for EsoPipe import
  - column_templates_discovered.txt: auto-clustered template patterns

Estimated effort: 1–2 hours of coding, 10 minutes to run.
Estimated value: VERY HIGH — this is the ground truth of your table style.

────────────────────────────────────────────────────────────────────────────────
SCRIPT 2: extract_documents.py
Purpose:  Build a bibliography of the PDFs that generated the conversations
────────────────────────────────────────────────────────────────────────────────

What it does:
  Scans tool turns (role='tool') for document title, author, and
  page-range patterns. Tool turns contain embedded PDF text that ChatGPT
  retrieved from uploaded documents. The first few lines often include
  metadata (title, author, journal, year).

Why it matters:
  Right now we know you studied alchemy in 187 conversations, but we don't
  know which of the ~1000+ academic articles in your PDF collection
  generated which conversations. Building this bibliography would:
  - Tell you which articles you have studied most thoroughly
  - Identify articles you have uploaded but never returned to
  - Enable the EsoPipe "source document" feature (table rows from PDFs)

Parsing approach:
  Tool turns often start with document metadata. Extract:
    - Lines matching "Author Name, 'Title', Journal, Year" patterns
    - DOI patterns: r'10\.\d{4}/\S+'
    - Page ranges: r'pp?\.\s*\d+[–-]\d+'
    - Chapter/section headings from the embedded text

Output:
  - documents table in DB: {id, conversation_id, inferred_title, inferred_author, year, doi}
  - bibliography.csv: sortable list of all source documents
  - conversation_document_map.csv: which conversation studied which document

Estimated effort: 2–3 hours. Heuristic-based, will need manual review.

────────────────────────────────────────────────────────────────────────────────
SCRIPT 3: chain_refinements.py
Purpose:  Trace the refinement chain for each table from first request
          to final accepted version
────────────────────────────────────────────────────────────────────────────────

What it does:
  Within each conversation, finds sequences of: table-request → table-response
  → refinement-request → refined-table-response → ...
  Links them into refinement chains and stores them.

Why it matters:
  The refinement chain is where your actual preferences live. The *first*
  table request captures your opening intent. The *final* accepted table
  (the one you stop refining) captures your actual preference.
  Comparing first-request columns to final-table columns reveals what
  you always end up adding or removing.

Output:
  - refinement_chains table in DB
  - refinement_patterns.txt: the 20 most common first→final changes
  - preferred_column_names.txt: columns that survive into final tables

Estimated effort: 2 hours. Requires careful heuristics to detect "acceptance"
(lack of further refinement request on the same table).

LLM TECHNIQUE 1: Prompt Style Profiler
Purpose:  Use an LLM to read 50 representative user turns and write a
          detailed description of your prompting style
────────────────────────────────────────────────────────────────────────────────

What it does:
  Samples 50 user turns across different request types and topics.
  Sends them to an LLM with a meta-prompt:
    "Analyse these 50 prompts written by a scholar. Describe their
    prompting style in detail: sentence structure, level of specificity,
    how they handle ambiguity, what they always include, what they
    never include, how they frame requests, what vocabulary they use
    for scholarly operations. Write a style guide that could teach
    someone to write prompts the way this scholar does."

Why it matters:
  The output would be a personal prompt style guide — something you could
  use to:
  - Evaluate whether future AI-generated prompts match your style
  - Train a custom GPT to prompt in your voice
  - Understand your own cognitive habits in externalised form
  - Identify prompting patterns you've drifted away from that worked well

How to run it:
  Extract 50 turns with: SELECT content FROM turns WHERE role='user'
  ORDER BY RANDOM() LIMIT 50
  Batch them into groups of 10 and send to the API.
  Aggregate the descriptions.

Estimated effort: 3 hours of scripting + API cost (~$2–5 depending on model).

VII.2  QUERYING THE DB DIRECTLY
  Any SQLite tool works. The most useful queries:

  -- All table requests with context
  SELECT c.title, t.content FROM turns t
  JOIN conversations c ON c.id=t.conversation_id
  JOIN request_types rt ON rt.turn_id=t.id
  WHERE t.role='user' AND rt.request_type='table'
  ORDER BY length(t.content) DESC;

  -- Find conversations about a specific topic
  SELECT c.title, c.date, c.turn_count FROM conversations c
  JOIN entity_mentions em ON em.conversation_id=c.id
  WHERE em.entity_id = 'thinker_bruno'
  GROUP BY c.id ORDER BY c.date;

  -- Get all assistant responses in a specific conversation
  SELECT turn_index, content FROM turns
  WHERE conversation_id = (SELECT id FROM conversations WHERE filename='GiordanoBruno.html')
  AND role='assistant' ORDER BY turn_index;

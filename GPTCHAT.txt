Great call. Here are 20 design decisions that will shape the build, organized by category. For each I'll give options and a recommendation.

I. Content & Data
1. What should the seed content actually be about?

The lessons, entities, and timelines need a subject. Three options:

A) Ikhwan al-Safa + Neoplatonism transmission — use the chat history you have (Ficino, Pico, Plotinus, etc.) as the actual entity/timeline data. The CS lessons explain how to build a database like this one.
B) Fully generic demo data — placeholder thinkers and texts (Aristotle, etc.) with no connection to your actual chats.
C) Mixed: generic CS lesson names + esoteric entities — lessons are abstract ("Graph Databases 101") but entities/timelines are from your real research domain.
Recommendation: A. Your chat history is rich, it makes the demo feel real and useful, and it directly demonstrates the "magical scholarship" framing. The Ikhwan al-Safa table you just showed me is perfect seed data.

2. How deep should the 3 seed lessons be?

A) Stubs only — title, summary, tags, 2–3 paragraphs, one code snippet. Fast to write, easy to extend.
B) Full worked lessons — 800–1200 words each, exercises with solutions, multiple code snippets, relational table embedded in the lesson itself.
C) One full + two stubs — one lesson is fully built as a reference implementation; the other two are shorter.
Recommendation: C. One full lesson shows what the format can do; two stubs keep the seed dataset manageable. Writing three full lessons adds ~2 hours for marginal value at the scaffold stage.

3. What should the 3 lessons actually teach?

A) Schema design, Graph queries, Search indexing — directly relevant to building this system.
B) NLP, text normalization, prompt design — more relevant to processing the chat transcripts you already have.
C) One from each: schema design + graph traversal + text search — covers the three main views of the app (Tables, Graph, Search).
Recommendation: C. Each lesson maps onto a page of the app, so clicking "Related lesson" on the Graph page leads to the graph-traversal lesson. Tightly coupled content/structure.

4. How should citations be handled?

Your context file has scholarly references (Carmela Baffioni, Nader El-Bizri, etc.). Options:

A) Citations as optional free-text strings — simple, no validation, easy to fill in.
B) Citations as structured objects {author, title, year, url?} — enables sorting/display but requires more seed data work.
C) No citations in seed data, but schema supports them — field exists in the type, left empty in seed data.
Recommendation: C unless you want to invest time in the data. The schema should be citation-ready, but don't block the build on populating them.

II. Technical Architecture
5. Graph library: which one?

A) Cytoscape.js — mature, huge API, handles large graphs, good layouts (cola, dagre, cose). Heavier (~300KB), somewhat imperative API.
B) Sigma.js + Graphology — designed for large graphs, WebGL rendering, more modern. More setup.
C) React Flow — React-native graph library, excellent DX, good for smaller graphs (<200 nodes). More "diagrammatic" look, less "network analysis" feel.
Recommendation: A (Cytoscape). Your prompt specified it and it's the best fit for exploratory knowledge graphs with filter panels. React Flow looks more like a flowchart tool.

6. Timeline: which library?

A) vis-timeline — powerful, battle-tested, but jQuery-era API, hard to integrate cleanly with React, large bundle.
B) Custom SVG/CSS timeline — full control, no extra deps, ~100 lines of code, but no zoom/pan out of the box.
C) react-chrono — lightweight React timeline component, clean API, supports vertical/horizontal, no zoom.
Recommendation: C (react-chrono) or B. vis-timeline integration with React is painful. A custom timeline gives you complete control over the esoteric styling and avoids wrestling with a non-React library. I'd go custom: it can be done in ~150 lines and look exactly right.

7. Search: which library?

A) MiniSearch — tiny (7KB), full-text with fuzzy matching, good TypeScript support, prebuilt index works offline.
B) FlexSearch — faster, more powerful, but more complex API, harder to serialize.
C) Fuse.js — simplest API, purely fuzzy (no full-text), smallest but least powerful.
Recommendation: A (MiniSearch). Best balance of power and simplicity. Precomputing the index as a JSON file at build time is straightforward.

8. Should there be a build-time search index generation script?

A) Yes — Node script writes /public/data/search-index.json at build time — requires running npm run index before npm run build, but loads instantly.
B) Build index at runtime in the browser on first load — simpler setup, tiny delay on first search.
C) Build index in a Web Worker — best UX, most complex.
Recommendation: B for simplicity. With only ~30–50 searchable items in the seed data, indexing takes <5ms in the browser. You can switch to A later if the dataset grows.

9. Data loading: Zod validation — when should it run?

A) At app startup, blocks render until validated — any bad data shows an error page before the user sees anything.
B) Lazy per-page — validate each dataset when its page first loads.
C) Dev-only validation — Zod only runs in development; production skips it for performance.
Recommendation: A. The prompt specifies a friendly error page listing invalid items, and the datasets are small. Adds ~10ms at most.

10. State management?

A) React Context + useReducer — no extra deps, sufficient for this app's needs (selected node, drawer open state, search query).
B) Zustand — tiny, ergonomic, better for complex cross-component state. Adds one dep.
C) URL state only — everything is a route param or query string. Shareable links for free.
Recommendation: A + C hybrid. Use URL params for selected entity/lesson (so links are shareable), React Context only for UI-only state (drawer open, search box value).

III. Design & UX
11. Overall visual aesthetic?

A) Dark academia — dark background, cream text, serif headings, deep burgundy/navy accents. Atmospheric, matches the domain.
B) Parchment/manuscript — warm off-white background, ink-brown text, gold accents. Like an illuminated manuscript meets a modern web app.
C) Clean academic journal — white background, black text, minimal color, emphasis on typography and spacing. Easier to read dense content.
Recommendation: B. The Tailwind colors I've already set up (parchment, ink) support this direction. Option A can look muddy when displaying tables; Option C is fine but generic.

12. Navigation layout?

A) Left sidebar (persistent, icon + label) — standard for data-rich apps, always visible, good for jumping between views. Takes ~60px of width.
B) Top navigation bar — more horizontal space for content, but you lose context when deep in a page.
C) Collapsible left sidebar — best of both, but adds complexity.
Recommendation: A. Your prompt specified left nav with icons. It's the right pattern for an 8-route app with a detail drawer on the right.

13. Right-side detail drawer — behavior?

A) Slides in over content — keeps main view visible, common pattern (VS Code, Notion).
B) Pushes content left — more disruptive but the entity stays fully in view beside context.
C) Modal dialog — simpler to implement, but breaks the browse-while-reading flow.
Recommendation: A. Overlay drawer is the most fluid pattern for "click a node, see details, keep browsing."

14. Mobile support?

A) Full responsive design — sidebar collapses to bottom nav on mobile, all views reflow. Adds significant CSS complexity.
B) Desktop-first with graceful degradation — works on mobile but isn't optimized. Some horizontal scroll in tables/graph.
C) Desktop-only explicit — add a "best viewed on desktop" banner on small screens.
Recommendation: B. The graph and timeline views fundamentally need space. Don't over-engineer mobile for a research tool that will likely be used on a desk.

IV. Feature Scope
15. Table explorer — how many curated tables?

A) 3 tables — "Key Thinkers", "Key Texts", "Transmission Timeline Summary" (derived from entity/timeline data).
B) 5–6 tables — add "CS Concepts Glossary", "Edge Types", "Lesson Index".
C) Dynamic: every entity type becomes a table — fully data-driven, scales automatically.
Recommendation: C. If each entity type (thinker/text/term/tool/concept) auto-generates a table, you get 5 tables for free from the seed data with zero extra work.

16. Graph — what interaction depth?

A) Click node → drawer opens with details + edges listed as text
B) Click node → drawer + highlight connected nodes in graph + "expand" button to show neighbors
C) Full graph exploration: filter by type, search nodes, expand/collapse clusters
Recommendation: B. Option A is too minimal; Option C is weeks of work. B gives real utility without scope creep.

17. Code snippets in lessons — what language support?

A) Syntax highlighting via a library (Prism.js or highlight.js) — proper coloring, ~30KB added.
B) Styled <pre> blocks with monospace font, no highlighting — zero deps, readable but plain.
C) Copy-to-clipboard only, no highlighting — simplest.
Recommendation: A. Prism.js is lightweight and the lessons need to be genuinely readable if this is meant as a teaching tool.

18. The /schema page — how technical should it be?

A) Pretty documentation page — visual cards showing each schema type with example JSON, edge type table, ID conventions.
B) Raw JSON schema display — shows the actual Zod schema output or JSON Schema.
C) Interactive schema explorer — click a type to see all instances, linked to live data.
Recommendation: A. The schema page should be readable by a researcher who isn't a developer. Pretty cards with example JSON, not raw schemas.

19. Exercises in lessons — show/hide solution toggle?

A) Simple accordion — click "Show Solution" to expand.
B) Blurred/obscured text — hover or click to reveal. More gamified.
C) No exercises in seed data — leave the field empty, schema supports it for future.
Recommendation: A. Simple accordion is standard and doesn't require extra interaction design. Blurred text is a cute trick but annoying when you actually want to read.

20. "Copy link" and sharing features?

A) Copy URL to clipboard button on every entity/lesson — since routes are URL-driven, this works automatically.
B) Export to JSON/CSV per entity — more useful for research workflows.
C) Both A and B
Recommendation: C. Copy link is trivial (1 line). CSV export on the tables page is genuinely useful for a researcher and TanStack Table makes it easy.

from fastapi import FastAPI
from fastapi.staticfiles import StaticFiles
from fastapi.responses import FileResponse
from pathlib import Path

app = FastAPI()

UI_DIR = Path(__file__).resolve().parent / "ui"

if UI_DIR.exists():
    app.mount("/assets", StaticFiles(directory=UI_DIR / "assets"), name="assets")

    @app.get("/")
    def index():
        return FileResponse(UI_DIR / "index.html")

    @app.get("/{full_path:path}")
    def spa_fallback(full_path: str):
        # Let API routes win; only fallback for unknown paths
        candidate = UI_DIR / full_path
        if candidate.exists() and candidate.is_file():
            return FileResponse(candidate)
        return FileResponse(UI_DIR / "index.html")

B) Make the exe start/stop cleanly without terminals
You already have esopipe2/desktop.py. The key improvements:

Choose an open port automatically.

A robust desktop launcher pattern:

import socket
import subprocess
import sys
import time
from pathlib import Path

def pick_port() -> int:
    s = socket.socket()
    s.bind(("127.0.0.1", 0))
    port = s.getsockname()[1]
    s.close()
    return port

def wait_ready(port: int, timeout_s: float = 10.0) -> None:
    start = time.time()
    while time.time() - start < timeout_s:
        try:
            import urllib.request
            urllib.request.urlopen(f"http://127.0.0.1:{port}/health", timeout=0.5)
            return
        except Exception:
            time.sleep(0.1)
    raise RuntimeError("Server did not become ready in time.")

def main():
    port = pick_port()

    # Start uvicorn as subprocess so PyInstaller onefile stays responsive.
    # Note: use python -m uvicorn to ensure module resolution
    server_cmd = [
        sys.executable, "-m", "uvicorn",
        "esopipe2.server:app",
        "--host", "127.0.0.1",
        "--port", str(port),
        "--log-level", "warning",
    ]

    creationflags = 0
    if sys.platform.startswith("win"):
        creationflags = subprocess.CREATE_NO_WINDOW

    server_proc = subprocess.Popen(server_cmd, creationflags=creationflags)

    try:
        wait_ready(port)

        url = f"http://127.0.0.1:{port}/"
        # Edge app mode window
        edge = r"C:\Program Files (x86)\Microsoft\Edge\Application\msedge.exe"
        args = [edge, f"--app={url}", "--new-window"]
        ui_proc = subprocess.Popen(args, creationflags=creationflags)

        ui_proc.wait()

    finally:
        if server_proc.poll() is None:
            server_proc.terminate()

if __name__ == "__main__":
    main()

Project Overview
EsoPipe is a computational humanities project that treats scholarly research as a data pipeline. The system provides a framework for transforming the informal materials scholars accumulate—PDF libraries, research notes, chat transcripts with AI systems, bibliographic lists, and analytical tables—into a structured, queryable knowledge archive.

The project was designed around a simple observation: modern scholars increasingly conduct research through conversations with large language models. Those conversations generate structured knowledge artifacts—tables, classifications, bibliographies, relational mappings—but the results are usually lost in chat history. EsoPipe captures those artifacts, normalizes them into relational data, and makes them explorable through a database, timelines, and knowledge graphs.

The result is a research environment where a scholar can move fluidly between narrative exploration, structured data analysis, and computational experimentation.

The worked domain for the current curriculum is the history of Neoplatonism and its transmission into Renaissance magic, but the architecture is domain-agnostic and can be applied to any humanities corpus.

Core Idea: Research as a Data Pipeline
Traditional humanities research produces essays and books. Computational research produces datasets and tools. EsoPipe attempts to unify these modes by treating scholarly inquiry itself as a pipeline.

The pipeline typically proceeds through several stages.

First, a scholar generates structured outputs from exploratory conversations with an LLM. These outputs might include bibliographic tables, conceptual taxonomies, or analytical summaries of texts.

Second, those outputs are normalized into relational tables. Entities such as thinkers, texts, concepts, places, and tools become rows in a database.

Third, relationships between entities are encoded as typed edges in a graph structure. For example:

Plotinus → influenced → Ficino

Corpus Hermeticum → translated_by → Marsilio Ficino

De Ente et Uno → engages_with → Neoplatonic metaphysics

Finally, these relationships can be explored through visual and computational interfaces such as timelines, graph visualizations, and relational browsers.

The result is a scholarly knowledge system rather than a collection of isolated notes.

What the Curriculum Teaches
The lessons on the site are structured as practical tutorials that teach core computer science ideas through humanities problems.

The goal is not to turn scholars into professional software engineers. Instead, the curriculum introduces a set of conceptual tools that allow researchers to work productively with modern AI systems and data infrastructures.

Key topics include:

Schema design

Scholars already design schemas whenever they categorize thinkers, define intellectual movements, or organize bibliographies. The curriculum formalizes that process into relational database structures.

Students learn how to design entity tables, relationship tables, and metadata fields that allow research materials to be queried programmatically.

Prompt engineering

LLMs become far more useful when prompts are structured to produce deterministic outputs. Lessons show how to design prompts that generate machine-readable tables, bibliographies, and analytical summaries.

Knowledge graph construction

Historical scholarship often involves tracing networks of influence, transmission, and interpretation. Knowledge graphs provide a formal way to encode these relationships.

Students learn how to represent intellectual history as a network of typed edges between entities.

Data pipelines

A research pipeline involves extracting information from PDFs, chat transcripts, and bibliographic lists, transforming that information into normalized structures, and loading it into a database.

The curriculum provides Python scripts and workflow templates for building these pipelines.

text: string

index: LinkableIndex

mode: 'plain' | 'md' (if you pass already-rendered HTML/markdown separately)

excludeRanges?: to skip code blocks / inline code

Tokenization approach (recommended):

Use a trie / Aho–Corasick style matcher over normalized text for multiword phrases.

Enforce word boundaries (\b) on both ends for single-word matches.

Prefer longest match first to avoid “Plotinus” inside “Neo-Plotinus” issues.

Safety rules (to prevent garbage linking):

Do not link inside code blocks, inline code, URLs, file paths

Minimum token length (e.g., >=4) unless it’s in an allowlist

Skip stopwords (“one”, “art”, “form”) unless capitalized or in a phrase match

Ambiguity handling:

If one match → normal link

If multiple matches → render as a link that opens a popover chooser (or skip, depending on your answer to Q6)

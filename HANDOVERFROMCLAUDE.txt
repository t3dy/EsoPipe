LLM (Large Language Model)
An LLM is a statistical system trained on massive amounts of text to predict the next word in a sequence. It does not “know” facts in the human sense; it generates language by modeling patterns in data. Its strengths are synthesis, paraphrase, stylistic control, and structured output when prompted carefully. Its weaknesses are hallucination, overconfidence, and inconsistency across long contexts. In your script, different LLMs are assigned different cognitive roles: one discovers research directions, others draft historical sections, another evaluates coherence and orders revisions. The key takeaway for your scholarly PDF pipeline is division of labor. Instead of asking one model to ingest 1,700 PDFs and “write something smart,” you assign constrained roles: extraction, summarization, comparison, verification. Treat the LLM as a probabilistic language engine that needs scaffolding, not as an autonomous scholar.

Model (Specific Version of an LLM)
A “model” is a particular trained instance with defined capabilities, token limits, and behavioral tendencies. Claude Sonnet, GPT-5.2, and Gemini 3 Pro are different systems with different training mixes and output styles. In the script, models are chosen strategically: one handles structured JSON discovery, one writes Enlightenment philosophy, one handles modern psychology, and a larger-context model edits. This is orchestration engineering: matching model strengths to task type. For your PDF pipeline, the lesson is to benchmark and specialize. Use a smaller, cheaper model for chunk classification or metadata tagging; use a larger model for synthesis across documents. Model choice becomes an architectural decision, not a default setting.

API and API Keys
An API is the programmatic interface that lets your Python script send prompts to a remote model and receive outputs. The API key authenticates you and tracks usage. Storing keys in a .env file and loading them at runtime is a security practice that prevents accidental exposure in version control. In the script, the pipeline refuses to run if required keys are missing. That “fail fast” pattern is a reliability discipline. In your scholarly pipeline, you will likely orchestrate Anthropic, OpenAI, vector databases, and maybe search APIs. Treat credential management as infrastructure. Build startup checks that verify embeddings models, database connections, and storage paths before processing thousands of PDFs.

LangChain
LangChain is a Python framework that standardizes how you prompt models, handle responses, and chain operations together. It abstracts provider differences so your code does not depend on raw HTTP calls. In the script, LangChain provides unified model wrappers for Anthropic, OpenAI, and Google, allowing the same workflow to call different systems. The takeaway for your project is abstraction. As you build AlchemyDB or a PDF ingestion system, avoid tightly coupling your code to one provider’s response format. Use a framework or your own wrapper layer so you can swap embedding models or summarizers without rewriting everything.

LangGraph
LangGraph is a workflow engine built on top of LangChain that lets you model LLM processes as directed graphs. Each node performs a function, and edges define transitions. It supports branching, loops, and parallel execution with state merging. In your script, the graph encodes: search → drafting fan-out → assembly → editor assessment → conditional revision loop → polish. This is declarative control flow. For your scholarly PDF system, this is crucial. You are not building “a prompt”; you are building a graph of operations: OCR → text cleaning → chunking → embedding → indexing → entity extraction → summarization → verification. Thinking in graph terms clarifies dependencies and failure recovery.

Pipeline
A pipeline is a structured sequence of transformations applied to data. Each stage has a clear input and output. In the article script, the raw idea becomes research queries, then research notes, then section drafts, then a unified manuscript, then an edited version, then a polished file. Pipelines enforce modularity and testability. In your PDF ingestion work, a pipeline might be: file normalization → metadata extraction → semantic chunking → embedding → vector indexing → relational tagging. The lesson is composability. Each stage should be testable independently so that when hallucination or corruption appears, you can isolate the source.

Node
A node is a single functional step in the graph. It might call an LLM, run a search, merge data, or write a file. In the script, nodes include Discovery, Deep Dive Search, Ancient Drafter, Enlightenment Drafter, Modern Drafter, Editor Assessment, and Polish. Nodes isolate responsibility. For your project, nodes might include “Extract Emblem Image,” “Generate Rich Profile JSON,” or “Cluster by Concept.” Node granularity should reflect logical responsibilities, not arbitrary code grouping.

Fan-Out and Parallel Drafters
Fan-out means multiple nodes execute concurrently using shared input but producing different outputs. In the script, three models draft different historical sections at the same time. This reduces runtime and separates intellectual domains. The reducer then merges results. In your scholarly system, fan-out could mean running entity extraction, citation detection, and thematic clustering simultaneously on the same chunk of text. The principle is parallel semantic processing followed by structured merging.

Reducer
A reducer merges concurrent outputs into a single state field. Without a reducer, one parallel result could overwrite another. In the script, _merge_dicts combines section drafts keyed by section name. This prevents race conditions. In your PDF database, you will need reducers when multiple processes annotate the same document—one adding entities, another adding keywords. Define deterministic merge rules so nothing silently disappears.

State (PipelineState)
State is a shared data object that travels through the graph. It holds research results, draft sections, revision counters, and final manuscript text. TypedDict enforces what fields exist and what types they have. In your scholarly system, state might include document metadata, extracted entities, embedding vectors, and validation flags. Structured state is how you prevent “string soup” chaos. You want explicit fields like raw_text, clean_text, entities, embeddings, verified_quotes.

Two-Pass Search
Two-pass search separates planning from retrieval. First, the model identifies what should be researched. Second, it retrieves or synthesizes information. In the script, the first pass generates search vectors; the second calls Tavily or generates a research brief. This is meta-reasoning: think before fetching. In your PDF system, first pass might detect conceptual gaps or required citations; second pass queries your vector database. This avoids blind retrieval and improves relevance.

Research Vectors
Research vectors are thematic categories guiding discovery. They structure coverage across historical eras. They prevent bias toward a single period. In your scholarly database, vectors might be “alchemy as artisanal practice,” “hermetic cosmology,” “economic networks,” or “visual symbolism.” Explicit vectorization ensures balanced retrieval rather than random semantic similarity.

Fallback Logic

Graph (in computing and workflow design)
A graph is a mathematical structure made of nodes (points) and edges (connections between points). In computer science, a graph represents relationships and flow. In a workflow engine like LangGraph, each node is a step (for example, “Search,” “Draft,” “Edit”), and each edge defines what happens next. Some graphs are linear (A → B → C). Others branch (A → B and A → C in parallel). Some include loops (A → B → C → B until condition met). In your script, the article pipeline is a directed graph: search feeds drafting; drafting fans out to three parallel nodes; results merge; the editor decides whether to loop back for revision. For your scholarly PDF pipeline, thinking in graphs clarifies design. OCR, chunking, embedding, entity extraction, clustering, and verification are not just “steps” — they are nodes in a directed graph with dependencies and possible feedback loops. Graph thinking prevents spaghetti code and makes scaling feasible.

Directed Graph
A directed graph (or digraph) is a graph where edges have direction. If A → B, that does not imply B → A. Workflow systems use directed graphs because processing flows forward. In the article script, research flows into drafting; drafting does not flow back into research unless the editor explicitly routes it. For your pipeline, direction matters. You do not want embedding to run before text cleaning. Directed edges encode logical ordering.

Conditional routing means the next node depends on a decision. In the script, the editor outputs either FINALIZE or REVISE. If REVISE, the graph routes back to a specific drafter node. In your PDF system, you might route a document to re-OCR if confidence is low, or skip reprocessing if quality thresholds are met. Conditional routing turns static workflows into adaptive systems.

README
EsoPipe — CS for Magical Scholarship
A data pipeline and interactive research site built from 249 ChatGPT conversations about Renaissance magic, Neoplatonism, Islamic esotericism, and related traditions.

Live Site
https://t3dy.github.io/EsoPipe/

What Is This
EsoPipe started as a question: what is actually in two years of scholarly AI conversations?

The answer turned out to be 249 conversations, 9,864 message turns, 3.16 million words of AI text, and 598 scholarly comparison tables — covering Plotinus, Ficino, al-Buni, Ikhwan al-Safa, Agrippa, Paracelsus, and dozens of related figures and texts. Rather than leave that material in a folder of HTML exports, this project mines it into a structured research tool.

The site also demonstrates that the skills used in building data pipelines — schema design, entity extraction, graph traversal, iterative prompt engineering — are the same skills needed for rigorous humanistic scholarship. The Lessons page makes that argument explicit.

Pipeline Scripts
mine_chats.py
Parses all 249 ChatGPT HTML exports into SQLite (esoteric_archive.db). Extracts entity mentions, request type classifications, and conversation metadata. Also produces topic_catalog.html and mine_report.txt.

python mine_chats.py --dir . --db esoteric_archive.db --esopipe-dir cs-magical-scholarship/public/data/
extract_tables.py
Re-reads raw HTML files directly (bypassing the database) to recover Markdown pipe-tables whose line structure the mining script's whitespace normalisation would destroy. Also captures HTML table elements. Outputs tables_mined.json.

python extract_tables.py --dir . --db esoteric_archive.db
Results: 941 raw tables found across 120 files, 598 passing quality filters (min 3 columns, min 2 rows, deduplicated by column fingerprint).

/maketable — Claude Code Skill
A Claude Code slash command (~/.claude/commands/maketable.md) that reads any PDF or text file and generates all five table templates for it, then optionally appends results to tables_mined.json and pushes to the live site.

The skill knows the full 13-type column vocabulary, the formatting rules, and the EsoPipe JSON schema.

Model memorizes quirks of your data rather than general patterns.

Data Leakage:

Curse of Dimensionality:
High-dimensional spaces behave strangely; distance metrics become less meaningful.

Silhouette Score:
Metric to evaluate clustering quality.

Centroid:
The center point of a cluster.

Embedding Drift:
If you change embedding model, cluster structure changes.

Schema Drift:
When new prompt types emerge that don’t fit old taxonomy.
